# CORD-01: Shared-Key Group Addressing

`draft` `optional`

A primitive for sending end-to-end encrypted, self-addressed events over Nostr among a
set of parties who share a symmetric key.

```json
{
  "kind": 1059,
  "id": "<id>",
  "pubkey": "<channel pubkey>",
  "tags": [["p", "<self ephemeral key>"]],
  "content": "nip44_encrypt({
    "id": "<id>",
    "pubkey": "<real author's pubkey>",
    "content": "nip44_encrypt({
      "id": "<id>",
      "kind": 9,
      "pubkey": "<real author's pubkey>",
      "content": "Hey chat!",
      "created_at": 1686840217,
    })",
    "kind": 13,
    "created_at": 1686840217,
    "tags": [],
    "sig": "<real author's pubkey signature>"
  })",
  "created_at": 1686840217,
  "sig": "<random, one-time-use pubkey signature>"
}
```

#z: [channelIds]
authors: [channelPubkeys]

This document specifies the **base cryptographic mechanism** of the Concord protocol,
independent of any application. It describes how a group of parties that share one 32-byte
secret can:

1. **Derive a common signing keypair** from that secret, so every holder computes the same
   public key (the *group address*), and
2. **Publish and read** standard [NIP-59](https://github.com/nostr-protocol/nips/blob/master/59.md)
   gift-wrapped events under that address, encrypted so that only key-holders can read them and
   addressed so that only key-holders can write to them.

Nothing here is specific to chat, communities, channels, members, or roles. Those are built
on top of this primitive in later CORDs. Here there are only **a shared key, a context, an
epoch, and the parties who hold the key.**

> **Frozen wire format.** The HKDF labels, the canonicalization byte layouts, the tag names,
> and the protocol version string in this document are **frozen**. Any change to a labeled byte
> orphans every prior event and forces a migration. Golden test vectors (Appendix A) pin these
> values.

> **Why derive an address from the key?** A naive design tags each event with a public
> group identifier so members can filter for it on relays. But then *anyone* can write that
> tag and force members to download and trial-decrypt junk — a denial-of-service. Here the
> group address is the **public key of a keypair derived from the shared secret**, and the
> event is *signed* by the matching secret key. Only a holder of the shared key can compute
> the signing key, so only a holder can emit an event that lands in a member's `authors`
> filter. Outsiders are shut out of the address entirely.

---

## 1. Roles of the keys

The primitive involves three keys.

| Key | Bytes | Role |
|---|---|---|
| **Identity key** | secp256k1 keypair | A party's normal Nostr `npub`. Signs the **seal** (authorship). Never on the wire in cleartext. |
| **Group key** (`GroupKey`) | 32 | The shared symmetric secret. Every party in the group holds it. Read access *is* possession of this key. |
| **Group signing key** (`GroupSigningKey`) | secp256k1 keypair | **Derived** (§3) from the group key plus a context id and an epoch. Every holder of the group key derives the *same* keypair. Signs the outer gift wrap and supplies the NIP-44 conversation key. Its **public key is the group address**. |

A *group* in this document is simply the set of parties who hold a given `GroupKey`. Each
holder still has their own identity key, which proves *who* authored each event from inside
the encryption.

Two further inputs select *which* derived keypair a given group key produces:

- **`ContextId`** — a raw 32-byte identifier naming the conversation/scope/topic this key is
  used for. Opaque here; later CORDs give it meaning. Never sent in hex inside the derivation.
- **`Epoch`** — a `u64` clock that advances each time the group key is rotated (CORD-02).
  Serialized **big-endian** inside the derivation; decimal string in tags.

The derived keypair's x-only public key is the **`GroupAddress`** — the value clients place
in a relay `authors` filter.

There is **no single-use ephemeral wire key**: the outer signer is the shared, member-
derivable `GroupSigningKey`.

---

## 2. The envelope

Every event is a standard NIP-59 three-layer construction. The only departure from vanilla
NIP-59 is that the **outer wrap is signed by the derived group key** (§3) rather than a random
one-time key.

```
rumor       = unsigned Nostr event (the real payload; kind names the action)
seal        = kind-13 event, content = NIP-44 v2 encrypt(conv_key, rumor.as_json()),
              SIGNED BY THE AUTHOR'S REAL IDENTITY KEY            (authorship)
gift wrap   = kind-1059 event, content = NIP-44 v2 encrypt(conv_key, seal.as_json()),
              SIGNED BY group_sk,  tags: ["p", group_pk], ["v", "2"]
```

`conv_key`, `group_sk`, and `group_pk` are the group signing key triple of §3 for this
`(group key, context id, epoch)`. The wrap's `pubkey` **is** `group_pk` — the relay-filterable
group address. The `["p", group_pk]` tag is NIP-59-conformant (it equals the signer here, since
the "recipient" is the whole group) and lets generic relays honor recipient-scoped deletion.

- **Kind 1059** (NIP-59 gift wrap) — every outer wire event.
- **Kind 13** (NIP-59 seal) — every inner authorship envelope.

### 2.1 Symmetric primitive

The envelope uses **standard NIP-44 v2** under the group conversation key:

- `conv_key = ConversationKey::derive(group_sk, group_pk)` — self-ECDH on the §3 group keypair.
- `seal(plaintext)  = NIP-44 v2 encrypt(conv_key, plaintext)`
- `open(content)    = NIP-44 v2 decrypt(conv_key, content)`

A wrong group key (hence a wrong `conv_key`) or a tampered payload fails the MAC. Because
`conv_key` is derived from the same shared keypair every holder computes, the seal and the
wrap use one key — there is no separate raw-key NIP-44 layer.

### 2.2 Rumor

The rumor is a normal **unsigned** Nostr event (NIP-59 rumor: it has an `id` but no `sig`).
Its authorship is proven by the seal that wraps it (§2.3), signed by the author's real
identity key. At this layer the rumor carries, at minimum:

- `kind` — the application action (defined by higher CORDs).
- `pubkey` — the author's real identity pubkey. **Must equal the seal signer** (§2.4).
- `created_at` — the send time in seconds.
- tags that **bind the rumor to its coordinate**:
  - `["context", <context_id hex>]` — the context id this event belongs to. **Required**, exactly once.
  - `["epoch", <epoch decimal>]` — the epoch. **Required**, exactly once.

Higher CORDs rename `context` to a domain term (e.g. `channel`) and add their own tags; the
binding requirement (§2.4) is what matters at this layer.

### 2.3 Layers in detail

- **Rumor** — unsigned; the real action. Has an `id` (the deterministic event hash) used for
  dedup and for references from other events.
- **Seal** (`kind 13`) — `content` = NIP-44 ciphertext of the rumor JSON under `conv_key`,
  `tags: []`, **signed by the author's real identity key**. This is the authorship proof; it is
  identical in shape to a NIP-59 seal.
- **Gift wrap** (`kind 1059`) — `content` = NIP-44 ciphertext of the seal JSON under `conv_key`,
  **signed by `group_sk`**, `pubkey = group_pk`. tags:
  - `["p", <group_pk hex>]` — NIP-59 recipient tag (here the group address itself).
  - `["v", "2"]` — the Concord protocol version. **Checked before any decryption**; an unknown
    version is rejected gracefully.

### 2.4 The binding triad

The three layers must bind to one coordinate. A receiver that finds any mismatch **drops the
event**:

1. The **wrap signer** (`group_pk`) must equal the group address derived for the context id and
   the exact epoch whose key produced `conv_key` (**strict equality**). This is implicit in §3:
   a holder of a different key produces a different `group_pk`, which simply won't match the
   `authors` filter, and whose `conv_key` won't open the content.
2. The **seal signer** (real identity) must equal the **rumor `pubkey`** — authorship binding.
3. The rumor's `context`/`epoch` tags must equal the context id and the exact epoch whose key
   decrypted the payload (**strict equality**, not a membership test).

Together these defeat insider replay/splice across context or epoch: a party holding the group
key cannot lift another party's sealed content into a different context, because the seal
signer plus the bound `context`/`epoch` pin it.

A **duplicate** `context` or `epoch` tag is rejected (it is constructable by any key-holder, so
first-match would be nondeterministic).

## 3. Key derivation (HKDF) — frozen

Every derivation in Concord is an HKDF-SHA256 expansion with no salt:

```
HKDF-SHA256(IKM, salt = ∅, info, L = 32)
info = utf8(label) || 0x00 || id32 || epoch_be
```

- `IKM` — the input keying material (here, the group key).
- `label` — an ASCII purpose string, no terminator.
- `0x00` — a single separator byte.
- `id32` — a raw 32-byte id (the context id), **never hex**.
- `epoch_be` — the epoch as `u64` big-endian (8 bytes); **omitted** where a derivation is
  epoch-independent.

`salt = ∅` means RFC-5869 with no salt (a zero-length salt; equivalent to 32 zero bytes for
HMAC-SHA256). `L = 32` (the expand never fails at this length).

> **`concord/v1` label prefix.** Every label uses the `concord/v1/…` prefix. The `v1` here is the
> *derivation* version, independent of the `v2` wire/protocol version string (§2.3): the HKDF
> construction is frozen and shared across CORDs. The labels are frozen — changing any byte
> orphans every derived coordinate. Appendix A pins the exact strings.

### 3.1 Deriving a secp256k1 scalar from HKDF

A 32-byte HKDF output is not necessarily a valid secp256k1 secret key. To turn one into a
scalar, use **reject-and-retry**: compute `HKDF(...info)`; if the result is not a valid scalar,
append a single incrementing counter byte to `info` and retry. The reject branch is ~2⁻¹²⁸ rare,
but the counter keeps the result deterministic and reproducible across implementations.

### 3.2 The group signing key (frozen)

The **group signing key** is a secp256k1 keypair every holder of the group key can derive,
whose x-only **public key is the group address** (the value clients query with `authors`) and
whose **secret key signs the outer gift wrap** and seeds the NIP-44 conversation key.

```
seed         = HKDF(IKM = group_key, label, id32 = context_id, epoch_be)   // 32-byte seed
group_sk     = scalar_normalize(seed, info)                                // §3.1 → valid secp256k1 sk
group_pk     = secp256k1_xonly_pubkey(group_sk)                            // 32-byte x-only; the wire address
conv_key     = NIP-44 v2 ConversationKey::derive(group_sk, group_pk)       // self-ECDH on the derived key
```

- `scalar_normalize` is exactly §3.1 (reject-and-retry with an appended counter byte) so the
  result is always a valid scalar and every holder computes the identical keypair.
- `group_pk` indexes the conversation on relays. Only a holder of `group_key` can compute
  `group_sk`, so **only members can emit events that match the address** — an outsider cannot
  spam a member's `authors` filter.
- `conv_key` is derived by self-ECDH of the group keypair with itself
  (`derive(group_sk, group_pk)`). Every holder arrives at the same `conv_key`; it encrypts both
  the gift-wrap content and the seal content (§2.1). The one derived keypair is both signer and
  encryptor; no separate raw-key envelope is needed.

The concrete `label` and `context_id` for a given application are assigned by higher CORDs.
This document fixes only the *construction*: `(group_key, label, context_id, epoch)` →
`(group_sk, group_pk, conv_key)`.

## 4. Open / verify procedure

To open a received gift wrap:

1. Read the `["v", ...]` tag and verify `v == "2"`. Reject an unknown version **before** any
   decryption.
2. Confirm the wrap `pubkey` is a `group_pk` you derived (i.e. it matched an `authors` you
   subscribed to). Select the corresponding `(group key, context id, epoch)`.
3. NIP-44-decrypt the wrap `content` under that `conv_key` to recover the **seal**.
4. Verify the seal's Schnorr signature (the author's real identity).
5. NIP-44-decrypt the seal `content` to recover the **rumor**.
6. Verify the rumor `id` (recompute the event hash).
7. Enforce the **binding triad** (§2.4): wrap signer = derived address, seal signer = rumor
   `pubkey`, rumor `context`/`epoch` = the coordinate you opened under.
8. Silently drop the event on any failure (wrong key, splice, forged seal, bad version).

---

## 5. Publishing and querying

### 5.1 Sending

1. Build the **rumor** (unsigned; `context`/`epoch` tags; `pubkey` = your real identity).
2. Derive the group signing key triple for `(group_key, context_id, epoch)` (§3.2).
3. **Seal** the rumor: NIP-44-encrypt the rumor JSON under `conv_key`, place it in a kind-13
   event, and sign with your **real identity** (local keys, a `NostrSigner`, or a NIP-46 bunker).
4. **Gift-wrap** the seal: NIP-44-encrypt the seal JSON under `conv_key`, place it in a kind-1059
   event with `["p", group_pk]` and `["v", "2"]`, and **sign with `group_sk`**.
5. Publish the wrap to the relays the group uses.

### 5.2 Querying

Query relays with:

```
kinds   = [1059]                         // every Concord wrapper is a gift wrap
authors = [ group_pk(group_key, context_id, epoch) for each retained (epoch, key) ]
until   = <oldest-known created_at secs>   // page older history; omit for the latest page
since   = <newest-held wire time secs>     // latest-page only, to skip re-pulling held events
limit   = <page size>                       // newest-first
```

Because the address is the wrap's **author**, a non-member cannot produce a matching event.
Process each returned wrap per §4. Then:

- **Dedup on the rumor id**, never the wrapper id — one rumor can ride multiple wrappers
  (re-broadcast, multi-relay copies, replays).
- **Order** deterministically by the rumor's authenticated timestamp, ties broken by rumor id.

### 5.3 Realtime subscription

Subscribe with one filter over `kinds = [1059]` and `authors = [` every group address you hold
across all contexts and epochs `]`, `limit = 0` (live tail). Route an arriving wrap by which
`group_pk` signed it, then open it per §4.

## 6. What is and isn't hidden

Relays and non-members see only NIP-44 ciphertext (kind-1059 gift wraps) authored by
**rotating per-epoch group keys**. They cannot read the payload, author identities, or the
context. The group address rotates per epoch (CORD-02), resisting — but not guaranteeing
against a global observer — correlation of a context's traffic. Because the address is the
wrap's author and only key-holders can compute the signing key, a non-member **cannot inject
events into a member's `authors` filter**.

**Not** hidden: that *some* encrypted traffic exists, its rough volume/timing, and your IP to
the relays you use (absent Tor/VPN).

> **Security tradeoff (inherited from the NIP-59 change).** A shared, member-derivable wrap
> signing key means **any member can issue a relay-level NIP-09 deletion** for the group address,
> and the group's traffic over an epoch is linkable by a watcher (the author pubkey is stable
> until the next rekey). This mirrors the tradeoff debated upstream
> ([nostr-protocol/nips#2396](https://github.com/nostr-protocol/nips/pull/2396)): per-conversation
> linkability and shared-key deletion power in exchange for spam/DoS prevention and `authors`-based
> filtering. Concord accepts it because (a) epoch rotation bounds linkability to one epoch, and
> (b) cooperative deletion is the app-level model in higher CORDs.
