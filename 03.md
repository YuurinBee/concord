# CORD-03: Rekeys and Refoundings

Non-ratcheted and Asynchronous Key rotation for Concord Communities for Post-Removal Secrecy.

**Rekeys**: allow Channels to rotate their encryption key to cryptographically sever a removed user (or many), authorized by a valid community administrator, single-channel rekeys are typically performed when removing someone's access to a specific channel, for example, removing a role ("`Tester`") with access to a private channel ("`#testers`").

**Refoundings**: are whole-Community Rekeys, rotating the base community key in addition to all relevant channels, this is done to remove someone absolutely from a Private Community, severing even the uppermost planes.

----

## 1. Rekey Blobs - The 3303 event

Key Distribution through Rekeys is handled through "Rekey Blobs", a Community-encrypted Kind 3303 package that delivers the fresh key to 120 participants per-event, there may be several Rekey Events for a sufficiently large community.

Rekeys have a given scope, a **Channel** (specific channel ID) or a **Community** (base rotation):
```
Scope       = Channel(channel_id) | CommunityRoot
id32(Scope) = channel_id[32]      | 0x00…00 (all-zeroes, never collides with a Channel ID)
```

Channels are independently keyed (cryptographically unrelated), a channel's key is never derived from the CommunityRoot, so a compromised channel exposes only *itself*, thus a channel can be rekeyed without touching community-wide access.

Because every key is independent, a rotation must name exactly *which one* it replaces: **Scope** handles this by binding to the blob's ciphertext (not just an outer tag), so a blob minted for one channel can never be replayed against another, neither accidentally or maliciously.

The Rekey Event:
```jsonc
{
  "kind": 1059,                          // gift wrap
  "pubkey": "<group_pk>",                // the rekey address (server-root-derived)
  "tags": [["p", "<group_pk>"]],
  "content": "nip44(conv_key, {         // seal
     "kind": 13,
     "pubkey": "<rotator real pubkey>",
     "sig": "<rotator real signature>",  // authorship + authority actor
     "content": "nip44(conv_key, {      // rumor
        "kind": 3303,
        "pubkey": "<rotator real pubkey>",
        "content": "[ {blob}, {blob}, … ]",   // per-recipient blobs (§4)
        "tags": [
          ["scope",      "<id32(Scope) hex>"],   // all-zero hex = ServerRoot
          ["newepoch",   "<decimal>"],
          ["prevepoch",  "<decimal>"],
          ["prevcommit", "<hex>"]
        ]
     })"
  })"
}
```

## 2. Receiving & Processing Rekeys

For receivers of the event, it's easy to detect whether or not you've been removed from the channel and/or community, at which point your client should either visibly remove you from the community, or should switch to using the new keys (a new epoch).

**Subscription**:
Rekeys can be received in real-time over the Community Root subscription, by precomputing the next rekey:
```jsonc
{
  "kinds": [1059],
  "authors": [
    // per channel you hold; the NEXT epoch's channel-rekey address:
    rekey_group_pk(server_root, channel_id, epoch + 1),
    // and the next base-rotation address:
    base_rekey_group_pk(prior_root, community_id, epoch + 1)
  ]
}
```

Each key blob has a **locator**, derived from the public key of the Rotator and Recipient, as well as the rotation scope and epoch:
```
locator = hkdf(ikm = rotator_xonly[32] || recipient_xonly[32],
               info = "concord/recipient-pseudonym" || 0x00 || scope_id[32] || epoch_be[8])
```

To check if you are part of a rekey, compute your locator and search for it within **all rekey events**, if it isn't in one, it may be in a consecutive event, you **must** validate the rekey came from a role-authorized administrator before accepting the event(s).
- If any event contains your locator, you can decrypt the new keys and shift to the new epoch.
- If all events do not contain your locator, you have been removed.

## 3. Refounding

A Refounding is a deep whole-Community rekey, performed when Banning a member from a Private Community, or when converting a Public Community to a Private Community:
- The Refounder prepares a compaction of the current Control Plane in to a new, slim representation; if the Refounder cannot reliably fold all Control Events in to a compaction, Refounding must be aborted.
- The CommunityRoot is rolled.
- The compacted Control Plane is republished under the new CommunityRoot. (ONLY after successful CommunityRoot roll publish confirmation).
- All channels relevant to the removed user(s) are rekeyed.

After a Refounding, the Control Plane is reduced back to its smallest size by trimming prior history, instead creating a "snapshot" of the prior Community Epoch's state; the last Control Plane state is simply rewrapped, so the signatures of prior authoritative members remain verifiable.

The compaction allows Community Rekeys to be faster and to remain asynchronous, allowing members to hop across dozens of epochs with thousands of Control Events without reprocessing the entire Control Plane each time.
